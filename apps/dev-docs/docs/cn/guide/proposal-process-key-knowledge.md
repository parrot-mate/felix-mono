# 提案流程中的关键知识

这篇指南用于说明在 PMate 中编写或评审 proposal 时需要掌握的核心知识。它不只是一个“文件结构清单”，而是要帮助贡献者理解每个 proposal 文档的职责、每类内容应该放在哪里，以及 proposal 在通过后应如何衔接到实现文档。

## 为什么需要这篇指南

PMate 的 proposal 采用四个必需文件：

- `product.md`
- `develop.md`
- `qa.md`
- `deploy.md`

这个结构本身是对的，但仅有结构还不够。贡献者还需要理解：

- 为什么要这样拆分内容
- 如何判断一段内容应该写进哪个文件
- 每个文件内部应该解释哪些关键概念
- proposal 通过后，如何转化为面向实现的文档

## 提案树结构速览

proposal 流程可以理解为一个两层树结构。

### 根层级

- Product
- Dev doc
- QA doc
- Deploy doc

### 第二层

每个根主题都应包含几个稳定的知识主题，而不是零散备注。

- Product:
  - 问题与背景
  - 用户价值与目标
  - 范围与非目标
  - 风险与开放问题
- Dev doc:
  - 技术设计
  - 仓库影响范围
  - 实现阶段拆分
  - 通过后的 guide 更新
- QA doc:
  - 什么是 unit test
  - 什么是 integration test
  - 什么是 e2e test
  - 如何编写可执行 Markdown 测试用例
  - 回归与边界场景
- Deploy doc:
  - 环境假设
  - 上线步骤
  - 回滚方案
  - 运行检查项

## Product

`product.md` 负责解释为什么这项工作值得做。

### 这里应该写什么

- 要解决的问题
- 哪类用户或角色会从中受益
- 成功标准是什么
- 哪些内容在范围内，哪些不在范围内
- 产品侧风险与不确定项

### 常见错误

不要把 `product.md` 写成纯技术实现方案。如果内容主要在讲包名、API、目录调整或测试框架，这些通常应该放在 `develop.md` 或 `qa.md`。

### 自检问题

可以问自己一句话：“如果产品负责人或评审只看这一份文件，是否能理解用户问题和预期结果？”

## Dev doc

`develop.md` 负责解释通过后的工作应如何实现。

### 这里应该写什么

- 技术设计
- 受影响的仓库与关键路径
- 实现阶段、依赖与执行顺序
- 关键取舍
- 必要时的架构图

### 设计和任务拆解的区别

这两个概念很容易混淆：

- 设计回答的是“系统应该如何工作”
- 任务拆解回答的是“应该按什么顺序完成工作”

二者都可以写在 `develop.md`，但它们不是同一种内容。

### 常见错误

不要把 `develop.md` 写成产品背景说明，也不要把它写成没有上下文的 todo list。开发者需要的是足够清晰的实现信息，而不是只有几个模糊动作词。

## QA doc

`qa.md` 负责解释 proposal 应如何被验证。这一部分最适合作为教学示例，因为很多人会混淆不同测试类型。

### 什么是 unit test？

unit test 用于在隔离环境中验证一个很小的行为单元。

例子：

- 一个函数
- 一个类的方法
- 一个依赖被 mock 的 React 组件

典型目标：

- 快速、精确地验证局部逻辑

典型边界：

- 数据库、浏览器、网络请求等外部系统通常会被 mock 或 stub

### 什么是 integration test？

integration test 用于验证多个模块组合后能否正确协作。

例子：

- API 路由和数据库层一起工作
- service 逻辑和队列或存储适配器一起工作
- UI 流程连接真实内部模块，但对外部依赖做受控处理

典型目标：

- 发现 unit test 不容易覆盖到的模块连接问题

典型边界：

- 测试会把多个真实组件连起来，但不一定覆盖完整系统的端到端流程

### 什么是 e2e test？

e2e test 用于从外部验证一个完整用户流程，尽量接近真实使用方式。

例子：

- 用户打开应用、登录、提交表单，并看到结果
- 操作者在接近真实环境中运行 CLI 命令并校验输出

典型目标：

- 证明这个功能从用户或操作者视角确实可用

典型边界：

- 测试会跨越 UI、API、存储、外部集成等多个系统层级

### 这三类测试如何配合

这三类测试不能互相替代。

- Unit test 提供快速的局部反馈。
- Integration test 负责发现模块边界联动问题。
- e2e test 用于验证真实流程。

一个高质量 proposal 应说明需要哪些测试层级，以及原因是什么。

### `qa.md` 还应该写什么

- 可执行的 Markdown 测试用例
- 前置条件
- 步骤化验证流程
- 预期结果
- 回归检查
- 边界场景
- 修复 bug 后的重跑要求

### 常见错误

不要只写“补测试”或“使用 vitest/playwright”。好的 QA 计划必须说明：测什么、在哪一层测、什么结果能证明改动正确。

## Deploy doc

`deploy.md` 负责解释如何安全地把改动带到真实环境。

### 这里应该写什么

- 环境假设
- 部署方式
- 上线步骤
- 回滚或恢复方案
- 发布后的运行检查

### Deploy 和 Dev / QA 的区别

- `develop.md` 关注实现设计
- `qa.md` 关注验证方式
- `deploy.md` 关注安全上线与运行信心

### 常见错误

不要把 deploy 当作最后补充的一小段。即使是文档或流程调整，也可能需要发布路径、说明更新或可见性检查。

## 提案通过后会发生什么

proposal 一旦通过，不应只停留在 `pmate-proposal` 仓库中。

当 proposal 对应的是实际功能实现或可重复执行的工作流时，关键的操作知识还应同步沉淀到 `pmate-mono` 的开发文档中：

- `apps/dev-docs/docs/en/guide/`
- `apps/dev-docs/docs/cn/guide/`

这份后续 guide 应说明：

- 通过了什么内容
- 实际执行时应该怎么做
- 哪些术语或决策最容易引起误解
- 实现时有哪些重要的仓库路径或命令

这样可以让 proposal 继续承担规划作用，同时把长期有效的实现知识沉淀到开发文档站点。
